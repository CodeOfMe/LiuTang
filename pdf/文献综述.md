# LiuTang 论文文献综述

> 本文档对 LiuTang 论文引用的全部 17 篇参考文献进行逐一综述，涵盖核心贡献、方法论、关键结果及其与 LiuTang 的关系。

## [1] carbone2015flink

**Paris Carbone, Asterios Katsifodimos, Stephan Ewen, Volker Markl, Seif Haridi, Kostas Tzoumas** | **2015** | **IEEE Data Engineering Bulletin**

Apache Flink 是一个开源的流批统一处理引擎，其核心理念是将实时流分析、连续数据管道、历史批处理和迭代算法（机器学习、图计算）统一在同一执行模型之下。论文详细阐述了 Flink 的架构：底层 runtime 基于 DataFlow 模型，将所有作业都视为流式数据流（batch 是有界流的特例）；通过 JobManager/TaskManager 架构实现分布式执行；采用基于 Chandy-Lamport 的分布式快照机制实现 exactly-once 语义。Flink 的时间语义支持事件时间与处理时间，窗口模型覆盖滚动、滑动和会话窗口。论文还讨论了 Flink 在迭代计算（如机器学习中的 Delta 迭代）上的优化策略。作为流处理领域的里程碑系统，Flink 奠定了现代流处理引擎的基本范式。

**与 LiuTang 的关系**：Flink 是 LiuTang 的核心对标系统。LiuTang 沿用了 Flink 的流批统一哲学和 DataFlow 执行模型，但采用纯 Python 实现而非 JVM，降低了部署复杂度。Flink 的水位线、窗口、状态管理等设计为 LiuTang 的对应特性提供了重要的设计参考。

## [2] beam2016

**Google Cloud Platform / Apache Software Foundation** | **2016** | **Apache Beam 官方文档与社区项目**

Apache Beam 是一个统一的批流数据处理编程模型和 SDK 框架，其前身是 Google Cloud Dataflow SDK。Beam 的核心贡献在于定义了一套与执行引擎无关的编程模型（Pipeline、PCollection、PTransform 三大抽象），允许用户编写一次代码后在 Flink、Spark、Google Cloud Dataflow 等多个 Runner 上执行。Beam 提出"批流统一"的核心理念：同一套 API 同时适用于有界数据（批）和无界数据（流），这与 Google Dataflow 论文 [15] 的思想一脉相承。Beam 在流式语义方面支持事件时间水位线和窗口化处理，但其语义覆盖深度依赖于底层 Runner。Beam 本身不包含执行引擎，而是一个可移植的 API 层。

**与 LiuTang 的关系**：Beam 的批流统一 API 设计理念与 LiuTang 高度一致。LiuTang 同样提供统一的 Flow/Stream API，并通过 mode 参数切换批处理与流处理。不同之处在于：Beam 是"写一次，跑在多个引擎上"，LiuTang 是"零依赖，跑在单机纯 Python 上"。Beam 的水位线和窗口模型为 LiuTang 的对应抽象提供了语义参照。

## [3] fragkoulis2020survey

**Marios Fragkoulis, Paris Carbone, Vasiliki Kalavri, Asterios Katsifodimos** | **2020** | **VLDB Journal (预印本)**

这篇综述系统梳理了流处理系统二十年的演进历程，将流处理系统分为两代：第一代（2000-2010）以 TelegraphCQ、STREAM、Aurora/Borealis 为代表，主要关注流式查询和复杂事件处理（CEP），但在查询语义上存在显著分歧；第二代（2011-2022）以 Storm、Spark Streaming、Flink、Google Dataflow 为代表，在乱序数据管理、状态管理、容错与高可用、负载管理、弹性伸缩和运行时重配置等方面取得了重大突破。论文详细讨论了乱序数据处理从简单丢弃到水位线机制的发展历程；状态管理从无状态到键分区间隔状态（Keyed State）的演进；容错机制从 at-most-once 到 exactly-once 的提升；以及弹性伸缩从手动配置到自动调整的趋势。该综述清晰刻画了流处理系统"从专用到通用、从脆弱到健壮"的技术演进脉络。

**与 LiuTang 的关系**：该综述为 LiuTang 的定位提供了完整的技术谱系。LiuTang 所涉及的水位线、事件时间窗口、键分区状态、检查点等特性，正是第二代流处理系统的核心能力。LiuTang 的贡献在于以纯 Python 零依赖的方式将这些能力民主化，降低了流处理系统的使用门槛。

## [4] faust2018

**Robinhood Markets (原 Robinhood 团队)** | **2018** | **GitHub 开源项目 (已归档)**

Faust 是一个基于 Python 的流处理库，灵感来源于 Kafka Streams，专注于与 Apache Kafka 集成进行实时数据流处理。Faust 以 Python 装饰器风格定义流处理拓扑（`@app.agent()` 装饰器），使用 asyncio 实现异步并发，支持键分区（key_by）、表（Table）状态存储和 RocksDB 持久化。Faust 采用"至少一次"（at-least-once）投递语义，事件时间支持有限（主要依赖处理时间），不支持水位线机制和事件时间窗口。Faust 的架构围绕 Kafka 主题展开，每个 agent 消费一个或多个主题，通过"分区-消费者"映射实现并行。项目已被归档（archived），不再活跃维护。

**与 LiuTang 的关系**：Faust 代表了 Python 流处理的早期尝试，但其流式语义覆盖不完整（缺少水位线、事件时间窗口、exactly-once），且强依赖 Kafka。LiuTang 在设计上与 Faust 形成对比：不依赖外部消息队列、提供完整的水位线和窗口支持、支持三种投递语义可切换、提供 Lambda/Kappa/Adaptive 架构模式。Faust 的归档也说明了纯 Python 流处理框架的可持续发展挑战。

## [5] streamz2019

**Matthew Rocklin 等社区贡献者** | **2019** | **GitHub 开源项目 (python-streamz/streamz)**

Streamz 是一个轻量级 Python 流处理库，核心设计理念是使用 Python 生成器和管道（pipeline）风格构建实时数据流处理管线。Streamz 以 `Stream` 类为基础抽象，通过 `.map()`、`.filter()`、`.accumulate()` 等方法链式组合算子，形成处理管道。它支持同步和异步执行，可与 Consume、Dask、Kafka 等集成。Streamz 的特点是极轻量（核心代码极小），适合原型开发和简单流处理任务。然而，Streamz 在流式语义方面覆盖率极低：不支持水位线、不支持事件时间、不支持窗口化操作、不支持状态管理和检查点机制、无 exactly-once 语义保障、无背压机制。Streamz 本质上是一个"无状态的流式管道编排器"，而非完整的流处理引擎。

**与 LiuTang 的关系**：Streamz 展示了 Python 流处理库的最简形态——管道式算子链——但同时也暴露了轻量级框架在语义完备性上的根本不足。LiuTang 同样追求轻量，但在语义覆盖上做了全面的工程投入（水位线、窗口、状态、检查点），这正是 LiuTang 相对 Streamz 的关键差异。

## [6] akidau2021watermarks

**Tyler Akidau, Edmon Begoli, Slava Chernyak, Fabian Hueske, Kathryn Knight, Kenneth Knowles, Daniel Mills, Dan Sotolongo** | **2021** | **PVLDB, Vol. 14, No. 12**

本文对水位线（Watermark）进行了系统性的定义、语义分析和实现比较。论文首先明确了水位线的本质：水位线是与流关联的时间完整性标记，告知处理器某一时刻之前的事件已全部到达。论文指出了水位线解决的五类核心问题：(1) 计算唯一正确答案（如通知触发）；(2) 推理数据缺失（如 dip detection）；(3) 对无限流的非增量时间子集做处理；(4) 安全及时地回收过期输入和中间状态；(5) 提供管线健康度的可靠信号。随后论文详细比较了 Flink 和 Google Cloud Dataflow 两种等价但实现截然不同的水位线机制：Flink 采用集中式 WatermarkGenerator 逐源生成水位线，沿数据流图传播；Dataflow 采用分布式水印跟踪器（watermark tracker），在引擎内部以细粒度跟踪每个操作的输入完成度。论文还讨论了水位线的语义属性（单调性、完整性保证）及其在窗口触发和状态回收中的角色。

**与 LiuTang 的关系**：水位线是 LiuTang 事件时间窗口机制的理论基础。LiuTang 实现了论文所述的两种核心水位线策略——单调水位线（monotonous）和有界乱序水位线（bounded out-of-orderness），并在此基础上构建窗口触发与允许延迟机制。本文对 Flink 与 Dataflow 水位线实现的对比分析为 LiuTang 选择更简洁的纯 Python 实现路径提供了参考。

## [7] chandy1985distributed

**K. Mani Chandy, Leslie Lamport** | **1985** | **ACM Transactions on Computer Systems, Vol. 3, No. 1**

这篇经典论文提出了分布式系统全局状态快照的 Chandy-Lamport 算法。算法核心思想：在分布式系统中，进程通过消息通道通信，每个进程记录自身状态并沿通道发送"marker"标记，接收到 marker 的进程也记录自身状态，最终通过合并所有进程快照和通道状态来还原全局一致性快照。该算法不需要暂停系统执行，能保证生成的全局快照满足一致性条件（即快照中的每个消息要么在发送方和接收方的快照中都出现，要么都不出现）。论文还将全局快照应用于稳定性质检测（如终止检测、死锁检测）和检查点/恢复机制。Chandy-Lamport 快照算法是分布式流处理系统容错机制的基石，Apache Flink 的检查点机制即为其直接实现。

**与 LiuTang 的关系**：LiuTang 的检查点机制继承了 Chandy-Lamport 的核心思想，但针对单机架构做了简化。在分布式场景中，Chandy-Lamport 算法需要处理跨节点的 marker 对齐问题（如 [16] CheckMate 论文所分析的），而 LiuTang 采用的纯 Python 单机模型天然避免了分布式 marker 协调的开销，使得检查点机制更加轻量高效。

## [8] geldenhuys2022phoebe

**Morgan K. Geldenhuys, Dominik Scheinert, Odej Kao, Lauritz Thamsen** | **2022** | **IEEE International Conference on Cloud Computing (CLOUD)**

Phoebe 提出了一种面向分布式流处理的主动式系统自调优方法。论文关注的核心问题是：流处理工作负载本质上是动态的，静态配置要么导致服务质量（QoS）下降，要么造成资源过度配置。Phoebe 的方法包含三个核心组件：(1) 并行性能分析（Parallel Profiling）——在同一作业的不同并行度下同时运行短期探针，收集延迟-吞吐量-恢复时间数据，构建 QoS 模型；(2) QoS 建模——将延迟、恢复时间与并行度之间的关系建模为目标函数；(3) 时间序列预测——利用历史负载数据预测未来工作量需求，使弹性伸缩决策更具前瞻性和持久性。实验表明 Phoebe 能在保持稳定服务的同时减少资源过度配置，且恢复时间规划确保了故障后的快速恢复。

**与 LiuTang 的关系**：Phoebe 关注的动态负载适应问题与 LiuTang 的 AdaptiveFlow 架构模式直接相关。Phoebe 通过时间序列预测和 QoS 建模来动态调整并行度，而 LiuTang 通过粘度（viscosity）参数在流式（η→0）与批处理（η→1）之间动态调节处理粒度。两者都试图解决"静态配置无法适应动态负载"的共性问题，但 Phoebe 聚焦于分布式系统的并行度调优，LiuTang 聚焦于单机场景下的批粒度调度。

## [9] liao2024flock

**Gang Liao, Amol Deshpande, Daniel J. Abadi** | **2024** | **ACM Conference**

Flock 是一个基于 FaaS（Function-as-a-Service）平台的低延迟流式查询引擎。论文的核心观察是：现有 Serverless 数据分析系统依赖 S3 等外部存储进行函数间数据交换，导致高延迟和高成本。Flock 的创新在于利用函数调用负载（invocation payload）直接传递数据，消除了对外部存储的需求。Flock 采用模板化方法动态创建云函数、使用函数组（function group）机制处理数据聚合和洗牌、支持 SQL 和 DataFrame API。实验表明，Flock 在成本上优于 Apache Flink 10-20 倍，延迟和吞吐量相当。Flock 还通过流式函数编排避免了冷启动开销，实现了亚秒级延迟。

**与 LiuTang 的关系**：Flock 代表了流处理在 Serverless 架构上的前沿探索，与 LiuTang 的单机纯 Python 路径形成了有趣的对比。Flock 利用 FaaS 的弹性免除基础设施管理，LiuTang 通过零依赖免除部署复杂度——两者都在追求"让用户专注业务逻辑"的目标，但选择了截然不同的技术路径。Flock 的函数组数据交换机制对 LiuTang 的进程内数据传递设计有参考价值。

## [10] bytewax2022

**Bytewax, Inc.** | **2022** | **GitHub 开源项目 (bytewax/bytewax)**

Bytewax 是一个以 Python 为接口语言、Rust 为执行引擎的流处理框架，定位为"Python-native 的流数据处理"。Bytewax 的核心架构采用 Timely Dataflow 模型（源自 Rust 的 timely 库），支持事件时间处理和基于纪元（epoch）的递增计算。其流式语义覆盖包括：滚动/滑动/会话窗口、键分区状态（map_state）、水位线的有限支持（基于进度跟踪）。Bytewax 提供 exactly-once 语义（通过 epoch 机制），但需要用户手动管理状态快照。在执行模型上，Bytewax 支持多进程并行，但不提供内置的分布式集群管理，需要手动组建。Bytewax 的主要优势在于 Python API 的友好性和 Rust 内核的性能，但需要 Rust 编译工具链。

**与 LiuTang 的关系**：Bytewax 与 LiuTang 在"Python 友好"的定位上高度相似，但技术路线不同：Bytewax 用 Rust 内核换取性能但引入了编译依赖，LiuTang 用纯 Python 追求零依赖但牺牲了极致性能。在流式语义方面，LiuTang 比 Bytewax 覆盖更全面（提供了允许延迟、多种状态类型、State TTL、投递语义可切换、架构模式可切换等 Bytewax 缺失或有限的特性）。

## [11] bartoszkiewicz2023pathway

**Michał Bartoszkiewicz, Jan Chorowski, Adrian Kosowski, Jakub Kowalski, Sergey Kulik, Mateusz Lewandowski, Krzysztof Nowicki, Kamil Piechowiak, Olivier Ruas, Zuzanna Stamirowska, Przemysław Uznański** | **2023** | **arXiv:2307.13116**

Pathway 是一个统一批流数据处理框架，核心由 Python Table API 和 Rust 分布式增量数据流引擎组成。Pathway 的原始驱动力来自物理经济（物流、供应链、IoT）的实时分析需求，这些场景要求低延迟响应、上下文感知分析和机器学习驱动的推理。Pathway 的技术特色包括：(1) 统一批流运行时，批流共享同一套增量计算引擎；(2) 基于差分数据流（differential dataflow）的增量计算，实现高效的变更传播；(3) 支持乱序数据的事件时间处理；(4) 流式迭代图算法（如流式 PageRank），这是工业级流处理框架中少见的特性。基准测试显示 Pathway 在批处理和流处理场景下均优于 Spark 和 Flink。

**与 LiuTang 的关系**：Pathway 和 LiuTang 都追求批流统一的 Python 友好框架定位。Pathway 通过 Rust 内核实现高性能，LiuTang 通过纯 Python 实现零依赖。Pathway 的流式迭代算法能力值得关注，但 LiuTang 在架构模式（Lambda/Kappa/Adaptive）和粘度控制方面提供了 Pathway 未涉及的新维度。两者的对比揭示了"高性能混合语言"与"纯 Python 零依赖"两种哲学路线的取舍。

## [12] carl2026serverless

**Natalie Carl, Niklas Kowallik, Constantin Stahl, Trever Schirmer, Tobias Pfandzeter, David Bermbach** | **2026** | **IEEE Conference (预印本)**

本文提出了"流函数"（Stream Functions）的概念，作为 FaaS 模型面向短生命周期、轻量级流处理任务的扩展。核心观察是：Serverless 函数擅长处理离散事件但无法维持状态，流处理引擎擅长处理长运行流但启动开销大——两者对"短运行、轻量级、有状态"的流处理任务都不高效。Stream Functions 将短流作为执行、状态和伸缩的基本单元，通过基于迭代器的接口实现事件间逻辑的无缝衔接，同时保留了 Serverless 的弹性和 scale-to-zero 能力。实验表明，在视频处理场景中 Stream Functions 比成熟流处理引擎减少约 99% 的处理开销。

**与 LiuTang 的关系**：Stream Functions 揭示了流处理和 Serverless 之间的语义间隙——FaaS 假设短运行无状态，流引擎假设长运行有状态。LiuTang 的 AdaptiveFlow 和粘度控制机制间接回应了这一问题：当 η→0 时，LiuTang 接近逐事件处理的 FaaS 模式；当 η→1 时，接近传统批处理引擎。LiuTang 的粘度连续谱可视为对 Stream Functions 所提问题的另一种解法。

## [13] gulisano2023semantic

**Vincenzo Gulisano, Alessandro Margara, Marina Papatriantafilou** | **2023** | **PVLDB, Vol. 16, No. 11**

本文从语义角度研究流处理引擎（SPE）中算子的重叠问题，证明了一个核心定理：所有主流 SPE 的常见算子（Filter、Map、FlatMap、KeyBy、Window、Aggregate、Join 等）都可以表示为单个极简 Aggregate 算子的组合。该 Aggregate 算子仅依赖 DataFlow 模型的核心概念——键分区（key-by）和基于时间的窗口——且每个窗口最多输出一个值。论文形式化地证明了这一结论，并通过实验验证了仅使用 Aggregate 算子的 SPE 与使用专用算子的 SPE 在性能上的差异。更有趣的是，当放宽 Aggregate 算子的约束（允许每窗口多值输出），其表达能力进一步提升，且性能开销可控。这一发现意味着：(1) 不同 SPE 之间的算子具有可移植性；(2) 为非 SPE 框架支持流处理提供了简明的需求规格。

**与 LiuTang 的关系**：本文为 LiuTang 的算子设计提供了理论根基。LiuTang 的算子集（map, filter, flat_map, key_by, window, aggregate 等）恰好覆盖了论文所识别的"流处理算子语义最小集"。理解所有算子可归结为 Aggregate 的组合，有助于 LiuTang 在未来优化内部实现——通过将多算子融合为更少的 Aggregate 实例来减少数据交换开销。

## [14] esteves2020aion

**Sergio Esteves, Rodrigo Rodrigues, Luís Veiga, Gianmarco De Francisci Morales, Marco Serafini** | **2020** | **arXiv:2003.03604 (会议版本发布于 USENIX ATC)**

AION 是一个构建在 Flink 之上的迟到事件处理方案，旨在解决流处理系统中窗口状态内存管理的难题。论文的核心问题：当事件时间数据出现迟到时，系统需要保持已关闭窗口的状态以接受迟到事件，但无限保持状态会导致内存溢出。AION 的解决方案包含三个核心机制：(1) 前瞻性缓存方案（Proactive Caching）——利用流处理语义预测哪些窗口状态即将被访问，提前从持久存储加载到内存；(2) 预测性清理方案（Predictive Cleanup）——基于迟到事件到达概率模型决定何时永久丢弃窗口状态，防止存储无限增长；(3) 内存-持久存储分层状态管理——将窗口状态在内存和 SSD/HDD 之间透明迁移。实验表明 AION 在保持高吞吐、低延迟、低迟到的条件下，将内存使用维持在可持续水平。

**与 LiuTang 的关系**：AION 关注的迟到事件与窗口状态管理问题是 LiuTang 允许延迟（allowed_lateness）机制的直接理论支撑。LiuTang 的水位线窗口触发后可配置延迟容忍期，延迟期结束后才清理窗口状态。AION 的分层状态管理思想对 LiuTang 单机场景下的内存-磁盘状态分层有参考价值，尤其是在长迟到容忍期导致内存压力的场景下。

## [15] akidau2015data

**Tyler Akidau, Robert Bradshaw, Craig Chambers, Slava Chernyak, Rafael J. Fernández-Moctezuma, Reuven Lax, Sam McVeety, Daniel Mills, Frances Perry, Eric Schmidt, Sam Whittle** | **2015** | **PVLDB, Vol. 8, No. 12**

本文提出了 Google Dataflow 模型，是现代流处理理论和实践的奠基之作。论文的核心论点是：面对无界、乱序、大规模数据集，必须抛弃"数据最终会完整到达"的假设，转而承认数据永不完整、旧数据可能被撤回，并在此假设下通过原则性抽象在正确性、延迟和成本之间做出有意识的权衡。Dataflow 模型提出两个关键抽象：(1) 窗口化——将无界数据切分为有限的时间片段（固定窗口、滑动窗口、会话窗口、自定义窗口）；(2) 触发器——声明窗口输出在何时、以何种重复策略被具体化（如水位线触发、允许迟到、早期触发）。论文还深入讨论了累加模式（Discarding、Accumulating、Accumulating & Retracting）对结果语义的影响，以及在 MillWheel 系统上的实现经验。

## [16] siachamis2024checkmate

**George Siachamis, Kyriakos Psarakis, Marios Fragkoulis, Arie van Deursen, Paris Carbone, Asterios Katsifodimos** | **2024** | **IEEE ICDE**

CheckMate 是首个系统性评估流处理数据流中检查点协议的实证研究。论文实现了三种检查点协议并进行了对比分析：(1) 协调式检查点（Coordinated Checkpointing）——即 Flink 采用的基于 marker 对齐的方案，所有算子协同完成全局一致性快照；(2) 非协调式检查点（Uncoordinated Checkpointing）——各算子独立进行检查点，无需 marker 协调，但恢复时需要重放日志并计算恢复线；(3) 通信诱导式检查点（Communication-Induced Checkpointing）——在非协调式基础上增加进程间控制消息来限制回滚范围。核心发现：在均匀分布的工作负载下，协调式检查点性能最优；但令人意外的是，在偏斜工作负载下，非协调式检查点不仅具有竞争力，甚至性能超越协调式检查点。此外，非协调式检查点天然支持循环数据流（cyclic queries），而协调式的 marker 对齐在循环流中无法完成。

**与 LiuTang 的关系**：CheckMate 对检查点协议的系统比较为 LiuTang 的容错机制设计提供了实证依据。由于 LiuTang 基于单机架构，不需要处理分布式 marker 协调的复杂性问题，因此可以采用更简化的检查点机制（接近非协调式检查点的轻量方案），同时绕开了 CheckMate 所揭示的偏斜工作负载下协调式检查点的性能瓶颈问题。

## [17] spenger2025holon

**Jonas Spenger, Kolya Krafeld, Ruben van Gemeren, Philipp Haller, Paris Carbone** | **2025** | **arXiv 预印本**

Holon Streaming 是一个面向全局聚合（Global Aggregation）的 exactly-once 流处理系统，核心创新是窗口化冲突自由复制数据类型（Windowed CRDTs）。论文指出：在传统流处理系统中，全局聚合要么在单个任务实例上计算（瓶颈），要么通过静态聚合树实现（单点故障、延迟由最慢路径决定、故障恢复时延迟飙升）。Holon Streaming 的解决方案是：(1) 引入 Windowed CRDTs——将 CRDT 的可交换、可结合、幂等性质与窗口化计算结合，使全局聚合可以在多个副本上并行计算并保证确定性收敛；(2) 去中心化协调——利用 CRDT 的代数性质设计高效的去中心化故障恢复算法；(3) 编程模型保证确定性——所有程序输出在相同输入下确定。实验表明，Holon Streaming 在全局聚合工作负载上延迟降低 5 倍、吞吐量提升 2 倍，在故障场景下延迟减少 11 倍。

**与 LiuTang 的关系**：Holon Streaming 展示了 CRDT 在流处理全局聚合中的应用前景，提出了"去中心化协调 + 确定性"的新范式。LiuTang 当前采用的是单机模型，全局聚合在单节点上完成，不存在分布式瓶颈问题。但如果未来 LiuTang 向分布式扩展，Holon 的 Windowed CRDTs 方案为全局聚合的水平扩展提供了重要参考。此外，Holon 的"确定性编程模型"理念与 LiuTang 的 exactly-once 交付语义在语义保证层面一致。
